# Architecture (Godot 4.6, project root = repo root)

Turn-based combat game. Dice carry a roll value + element; player swaps/rotates dice to set up an attack, then ends turn to resolve combat against a monster.

## Autoloads (singletons)
- `Swatch` (Globals/swatch.gd) ‚Äî color constants + `from_name(key)`. `ELEMENT_COLOR`, `NEON_COLOR`, `HALF` indexed by Element.
- `GlobalSignal` (Globals/globalsignal.gd) ‚Äî global signal bus (see Signals).
- `CurrentRoll` (Globals/current_roll.gd) ‚Äî shared combat numbers + per-turn operators.
- `Constants` (Globals/rollables.gd, `class_name Rollables`) ‚Äî `enum RollIndex {BASE,MULT,ANTI,ANTI_TYPE}`, `enum Element {RED,GREEN,BLUE,WHITE}`. Most gameplay classes `extends Rollables`, so refer to enums bare (`RollIndex.BASE`) or as `Constants.RollIndex`.
- `CombatState` (Globals/combat_state.gd) ‚Äî the FSM driving the round.
- `View` (Globals/View.gd) ‚Äî shared camera state for 2D-projected 3D shapes (cubes). Not gameplay.
- `Encounter` (Globals/encounter.gd) ‚Äî gauntlet selector: `monster_list` (PackedScenes) + `current_monster_order`; `next_monster` is a getter returning `monster_list[order % size]`. The monster spawner reads `next_monster`.
- `Screenshot` (Globals/screenshot.gd) ‚Äî debug-only: F12 saves the viewport to `screenshots/` (gitignored) for the ui-spec ¬ß8 checklist. Not gameplay; editor/debug builds only (res:// is read-only when exported).
- `Combatants` (Globals/combatants.gd) ‚Äî live combatant registry: `player` + `monster` node refs. Entities **self-register** in `_ready` (`Combatants.player/monster = self`) and clear in `_exit_tree` (guarded `== self`, so a respawn's new monster isn't clobbered by the old one's clear). Lets the FSM read both HPs without scanning `round_participants`. Refs, not NodePaths (paths go stale on respawn/reparent).
- `RunLog` (Globals/run_log.gd) ‚Äî run history logger. The rework controller drives it (`begin_run`/`begin_fight`/`begin_round`/`record_action`/`record_result`/`end_fight`/`end_run`); `end_run` appends **one JSON line per completed run** to `user://run_log.jsonl`. Per-round record = `start` dice `[[value,element],√ó3]`, `monster_roll` (that turn's pattern), `hp_before`/`hp_after` (both sides), `action` (`swap`/`rotate`/`pass`, swap carries `rerolled`), `after` dice, `deal`/`take` (actual HP deltas). Fight fields (`monster`/`max_hp`/`hp_in`/`hp_out`) + run fields (`outcome`/`died_to`/`start_hp`/`final_hp`) mirror `balance_sim` for real-play-vs-sim comparison. Local-only, no upload; abandoned runs (no `end_run`) aren't written.

## Combat flow (CombatState FSM)
`INITIAL ‚Üí ROUND_START ‚Üí PLAYER_PLANNING ‚Üí TURN_RESOLVING ‚Üí PLAYER_ATTACK ‚Üí MONSTER_ATTACK ‚Üí (WIN | LOSE | back to ROUND_START)`. Each enter handler waits a 1s timer. `ROUND_START` calls `round_start()` on every node in group `round_participants` (sorted by `round_start_priority`). Player ends planning via `CombatState.end_player_turn()` (called from swap/rotate).

**Death is decoupled from attacks.** Every state ends through one gate, `_advance(next_state)`, which reads `Combatants.player.hp` / `Combatants.monster.hp` (via `_is_dead()` + `is_instance_valid`): player ‚â§0 ‚Üí LOSE, else monster ‚â§0 ‚Üí WIN, else `next_state`. So a kill from ANY source ‚Äî a plain attack, a mid-attack hit, or a future round-start DoT ‚Äî is caught at the next boundary, not just after an attack. Player is checked first, so a mutual kill resolves to LOSE for now (the tie "both win" / `RETREAT` / fight-start-gate model is designed but **parked** until effects exist ‚Äî see HISTORY 2026-06-25). The old vestigial `CHECK_DEFEAT` state + inline `_is_monster_dead`/`_is_player_dead` group-scans are gone; `State.gd`'s name array mirrors the enum so it stays index-aligned.

The FSM is **not** self-starting: it's kicked by `CombatState.start()` (reset to INITIAL ‚Üí ROUND_START) on combat load ‚Äî `combat_rework.gd._ready()` does this for the rework (legacy `combat_ui.gd` did it for the old scene). A proper scene-start/run-reset is still a TODO. **WIN drives a controller-owned gauntlet respawn**, NOT the FSM: `combat_rework._on_victory()` (fired `call_deferred` from `_on_state_changed` on entering WIN, so we don't re-enter the FSM mid-transition) waits a beat, advances `Encounter.current_monster_order`, `_respawn_monster()` (free old + `_spawn_monster` the next), then `CombatState.start()`s the next fight ‚Äî **player + HP persist** (only the monster is swapped). Last monster cleared ‚Üí stays on victory (run-complete flow is a later step). **LOSE is still terminal** (retreat/run-reset later). The FSM's `_on_win`/`_on_lose` are minimal (just a beat) ‚Äî no scene-specific work, since the old `monster_spawner` group doesn't exist in the rework and crashed. `Encounter.current_monster_order` is reset to 0 in the controller's `_ready` so each run starts at the first monster.

## Roll data (on CurrentRoll)
- `current_roll_list` / `current_monster_roll_list` = `[base, mult, anti, anti_type]`.
- Player: damage per hit = base (`current_roll_list[0]`), number of hits = mult (`current_roll_list[1]`).
- The old **mutating** combat path ‚Äî `anti_operator()`/`player_attack()`/`monster_attack()` + the `turn_end`/`monster_turn_end`/`monster_damage_operator` wrappers ‚Äî was **removed 2026-06-30**. `compute_outcome` (resolve) + `combat_state._apply_attack` (apply) replace it; `CurrentRoll` is now a resolver + shared roll-data store, not a combat executor.
- `initial_roll` / `initial_monster_roll` snapshot pre-anti values (used by `get_reduced_roll` for announcements).
- `compute_outcome(player_roll, monster_roll)` ‚Äî **pure** resolver (no side effects): applies anti both ways + base√ómult on copies, returns `{player/monster:{per_hit,hits,total,blocked,misses}}`. The **single** resolver ‚Äî the FSM uses it in `_on_turn_resolving` (‚Üí `CombatState._outcome`), the preview UI uses it, and the effect pipeline hooks here. `current_roll_list` is published by the rework controller on commit; `current_monster_roll_list` by `monster.update_roll` at round start.
- `next_pattern` (a `Pattern`) ‚Äî the upcoming round's pattern, published by `monster.update_roll`; drives the lookahead label/halo.

The FSM's `_apply_attack(target, side, hit_signal, miss_signal)` is a staggered coroutine (`attack_stagger`, default 0.3s): a miss loop then a hit loop, applying `side.per_hit` straight to the target's `Hp` (`take_damage`) and emitting the per-hit signals for juice only; it bails the instant the target is dead (no over-kill). It emits the per-hit signals (`player_attacked`/`player_missed` + monster twins) for **juice only** ‚Äî damage is applied directly to `Hp`. The lean monster (data set) does **not** connect `monster_hit` (would double-apply); the lean player never had a damage handler.

## Signals (GlobalSignal)
- `player_attacked` ‚Äî once per surviving hit, emitted by `combat_state._apply_attack` (juice only; damage applied directly to `Hp`). Legacy `monster.gd:monster_hit()` (fat monster, `data` null) still subtracts `player_damage` ‚Äî dormant in the rework.
- `player_missed` ‚Äî once per hit lost to anti (no HP change).
- `player_attack_finished` ‚Äî after the loops.
- `monster_attacked` ‚Äî once per surviving hit, emitted by `combat_state._apply_attack` (juice only). Legacy `player_character.gd:player_hit()` subtracts `monster_damage` ‚Äî dormant in the rework (lean player doesn't connect it).
- `monster_missed` ‚Äî once per monster hit lost to anti.
- `monster_atack_finished` (note the typo) ‚Äî fires once after the monster loops; `monster.gd:_announce_attack()` and `player_character.gd:announce_damage_taken()` log here so announcements fire once, not per hit.
- `updated_roll` ‚Äî dice/roll changed; player_character, slime current_roll, etc. refresh.
- `announced(String)` ‚Äî `announce.gd` appends to combat log.
- `swap_started(Dice)` ‚Äî `mouse.gd` reparents the dragged die.
- `preview_set(String)` / `preview_clear` ‚Äî hover preview: `swap.gd`/`rotate.gd` emit a hypothetical readout while dragging; `announce.gd` shows it, reverts on clear.

## Preview UI
- `announce.gd` (on `Anouncement`) is a dual readout: PREVIEW (`Deal X  Take Y` from `compute_outcome`, recomputed on `updated_roll` via `CONNECT_DEFERRED` so the player roll refreshes first) vs LOG (combat log). A horizontal swipe *started on the label* toggles modes.
- Hover preview: `player_character.preview_rotate(src,tgt)` (single value) / `preview_swap(src,tgt)` (`Deal X~Y` range over the 6 reroll values ‚Äî swap reroll is unpreviewable by design, so it shows stakes not result). `swap.gd`/`rotate.gd` `_process` polls the hovered zone during a drag and emits `preview_set`/`preview_clear`.
- `Pattern` (character/monster/pattern/pattern.gd) is a Resource: `type` (`enum Type {HEAVY,FLURRY,GUARDED,SPIKE}` ‚Äî authored role) + `base/mult/anti/anti_type`. `next_pattern.gd` (on `NextPattern` label) shows the next roll as `Base: N  Mult: N  Block/Miss: N` (Block if `anti_type==BASE`, else Miss). Role/type is meant to be conveyed by a halo color (deferred).

## Damage numbers
`damage_number_zone.gd` (on a `DamageNumberZone` Control with a `DamageNumberLabel` child, `class_name DamageNumber`) duplicates the child and pops it at a random point in its rect. Pop variants on the label (block when `blocked > 0`, else number; one MISS per `*_missed` signal; each has a `_monster` twin that tweens down via `pop_show_monster()`):

- `pop_show_block(original, blocked)` ‚Äî `base` (damage/hit) was reduced; shows two lines: `Blocked` / `original -blocked`.
- `pop_show_miss()` ‚Äî for each hit lost to `mult` reduction. e.g. mult 8‚Üí5 = 5 damage pops + 3 MISS pops.

- `hp.gd`: `max_hp`/`current_hp` are `@export` (set per monster in inspector). Setters guard `if label:` because `@export` assignment fires before `@onready var label`. HP **persists across a gauntlet** ‚Äî only the monster is freed/respawned between fights; the player node survives.
- Monster pattern cycling: `round_start()` calls `update_roll()` *before* `current_round += 1`, so round 1 uses `pattern[0]` (no skip).
- Autoloads `CombatState` and `Encounter` persist across scene reloads, so run-state resets (FSM, gauntlet order) are manual ‚Äî currently in `combat_ui.gd` `start()` and the win/lose handlers.
- **Don't use `:=` on autoload method calls.** `var o := CurrentRoll.compute_outcome(...)` fails to parse on a cold `.godot/` cache ("Cannot infer the type of 'o'") ‚Äî GDScript's analyzer doesn't reliably resolve a method's return type through an autoload singleton until the class registry is built. A warm cache hides it, so it can pass on one machine and break after another deletes `.godot/`. Annotate explicitly instead: `var o: Dictionary = CurrentRoll.compute_outcome(...)`. (Fixed at `player_character.gd` preview_rotate/preview_swap and `announce.gd._update_preview`.) When such a script fails to compile, scenes that instance its `class_name` (e.g. `PlayerCharacter`) can hard-crash the editor on scene-restore (null script deref) ‚Äî open in Recovery Mode to break the loop.
- **The bash sandbox mount can serve a STALE / truncated view of a file.** Seen 2026-06-12: `wc`/`cat`/`py_compile` reported `tools/balance_sim.py` truncated mid-line at 14701 bytes while the editor (Read tool) had the full, correct file ‚Äî and the mount stayed frozen at that byte count even after a fresh write. A prior session misread git the same way. **The Read/Write/Edit file tools are the source of truth for file contents; do NOT trust the shell over them when diagnosing "truncation" or "corruption."** A fresh session (or just re-running later) re-syncs the mount. Note: code *executed* via bash reads the (possibly stale) mount, so if a run fails on "truncation" the file is probably fine ‚Äî verify with Read, don't rewrite blindly.
